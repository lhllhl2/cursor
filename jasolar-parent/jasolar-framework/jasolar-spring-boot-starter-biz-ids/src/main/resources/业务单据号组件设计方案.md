# 业务单据号组件设计方案

## 1. 背景需求

在微服务架构中，各个业务系统需要生成符合特定规则的业务单据号。这些单据号需要满足以下要求：

- **规则定制**：不同业务类型有不同前缀规则
  - 如：供应商注册邀请单：1+01+YYYYMMDD+五位流水号
  - 供应商准入单：1+02+YYYYMMDD+五位流水号
  - 等等...
- **极致性能**：支持高并发请求
- **稳定可靠**：保证单据号不重复
- **高可用性**：系统故障时能够优雅降级

现有技术栈包括PostgreSQL和Redis，需要基于这两种技术实现单据号生成服务。

## 2. 设计方案

### 2.1 整体架构

采用"**Redis优先 + PostgreSQL备份**"的混合架构：

架构分为两个核心部分：
1. **IBizIdGenerator类库**：供各微服务直接引用，封装单据号生成逻辑
2. **SerialNumberGeneratorService**：独立微服务，负责持久化和故障恢复

### 2.2 工作流程

1. **正常流程**：
   - 直接通过Redis的INCR命令原子递增获取流水号
   - 根据业务规则格式化为完整单据号

2. **降级流程**：
   - Redis故障时，自动降级调用SerialNumberGeneratorService服务
   - 服务从PostgreSQL获取带"安全步长"的流水号

3. **持久化流程**：
   - SerialNumberGeneratorService定期将Redis计数器同步到PostgreSQL
   - 确保系统重启时能够恢复正确状态

### 2.3 关键设计点

1. **日期隔离**：每天重置流水号，确保长度可控
2. **跳步机制**：故障恢复时增加跳步(+100)，避免重复
3. **溢出处理**：当流水号超过99999时告警并处理
4. **职责分离**：类库负责高性能生成，服务负责持久化
