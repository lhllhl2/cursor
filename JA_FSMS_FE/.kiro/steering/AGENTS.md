# AI 协作指南

## 工作模式

### 方案建议型

在执行任何代码修改前，AI 应该：

1. **分析问题**：详细说明当前情况和需要解决的问题
2. **提供方案**：给出 2-3 个可行方案，每个方案包括：
   - 方案描述
   - 优点和缺点
   - 对项目的影响
   - 推荐理由（如果有推荐）
3. **等待确认**：等待用户选择方案后再执行
4. **执行并汇报**：完成后详细说明做了什么修改

## 沟通风格

### 详细解释

AI 在沟通时应该：

- **说明背景**：解释为什么需要这样做
- **解释决策**：说明选择某个方案的原因
- **列出影响**：说明修改会影响哪些文件或模块
- **提供上下文**：引用相关的项目规范或最佳实践

### 示例

**❌ 简洁风格（不要这样）：**
> 已修复 API 类型错误。

**✅ 详细风格（应该这样）：**
> 发现 `getUserListApi` 的返回类型不符合 api-standards.md 规范。
> 
> **问题**：当前返回类型为 `any`，缺少类型安全。
> 
> **方案 1**：使用现有的 `UserInfo` 类型
> - 优点：复用已有类型，保持一致性
> - 缺点：如果列表返回的字段与详情不同，可能不够精确
> 
> **方案 2**：创建新的 `UserListItem` 类型
> - 优点：类型更精确，符合实际返回数据
> - 缺点：增加类型定义，需要维护
> 
> **推荐**：方案 2，因为列表和详情的字段通常不完全相同。
> 
> 你希望使用哪个方案？

## 决策权限

### 必须询问的情况

AI 在以下情况下**必须停下来询问**，不能自行决定：

1. **多个同等合理的方案**
   - 有 2 个以上技术上都可行的实现方式
   - 各有优缺点，没有明显的最佳选择

2. **业务逻辑不明确**
   - 需求描述模糊或有歧义
   - 不确定用户的真实意图
   - 涉及业务规则的判断

3. **可能影响其他模块**
   - 修改会影响多个文件或模块
   - 可能破坏现有功能
   - 需要同步修改其他相关代码

4. **偏离项目规范**
   - 现有代码不符合 steering 规范，但大规模修改风险高
   - 需要在"遵循规范"和"保持稳定"之间权衡

5. **架构或设计变更**
   - 涉及目录结构调整
   - 需要引入新的依赖或工具
   - 改变现有的架构模式

6. **性能或安全考虑**
   - 可能影响性能的优化
   - 涉及安全相关的修改
   - 需要权衡功能和性能

### 可以自主执行的情况

以下情况 AI 可以直接执行（但仍需详细说明）：

1. **明显的错误修复**
   - 语法错误、拼写错误
   - 明显的类型错误
   - 导入路径错误

2. **严格遵循规范**
   - 按照 api-standards.md 编写 API
   - 按照 structure.md 组织文件
   - 使用 tech.md 中规定的技术栈

3. **用户明确指示**
   - 用户已经说明了具体的实现方式
   - 需求清晰无歧义

## 错误处理

### 遇到问题时

1. **说明问题**：清楚描述遇到了什么问题
2. **分析原因**：解释可能的原因（2-3 个）
3. **提供方案**：给出解决建议
4. **等待指示**：不要自行尝试多次，询问用户希望如何处理

### 示例

**遇到错误时应该这样：**
> 执行 `pnpm dev:budget` 时遇到编译错误。
> 
> **错误信息**：
> ```
> Type 'string' is not assignable to type 'number'
> ```
> 
> **问题分析**：
> `pageNo` 参数在 `getReportListApi` 中被定义为 `number`，但调用时传入了字符串类型的路由参数。
> 
> **可能原因**：
> 1. 路由参数默认是字符串，需要转换
> 2. API 类型定义可能不正确
> 
> **解决方案**：
> 1. 在调用前将字符串转为数字：`Number(route.query.pageNo)`
> 2. 修改 API 类型定义接受字符串或数字：`pageNo: string | number`
> 
> **推荐**：方案 1，保持类型严格性，在调用处转换。
> 
> 你希望使用哪个方案？

## 代码审查

### 完成后的汇报

每次完成代码修改后，AI 应该提供：

1. **修改清单**：列出所有修改的文件
2. **修改说明**：每个文件做了什么改动
3. **验证建议**：建议如何验证修改是否正确
4. **注意事项**：是否有需要注意的地方

### 示例

**完成后应该这样汇报：**
> ✅ 已完成报表列表 API 开发
> 
> **修改文件**：
> 1. `apps/web-budget/src/api/report/payment.ts` - 新建
> 2. `apps/web-budget/src/api/report/index.ts` - 更新导出
> 
> **详细说明**：
> 
> **1. payment.ts**
> - 创建 `ReportApi` namespace，定义了查询参数和响应类型
> - 定义 `enum API` 包含 3 个路径常量
> - 实现 3 个 API 函数：
>   - `getReportListApi` - 获取列表（支持分页）
>   - `getReportDetailApi` - 获取详情
>   - `exportReportApi` - 导出报表（返回 blob）
> - 所有函数都遵循 api-standards.md 规范
> 
> **2. index.ts**
> - 添加 `export * from './payment'` 统一导出
> 
> **验证建议**：
> 1. 运行 `pnpm check:type` 确保类型检查通过
> 2. 在组件中导入测试：`import { getReportListApi } from '#/api'`
> 3. 调用 API 查看网络请求是否正确
> 
> **注意事项**：
> - 导出接口返回 blob 类型，需要在组件中处理文件下载
> - 分页参数 `pageNo` 从 1 开始，不是 0

## 项目规范优先级

在做决策时，遵循以下优先级：

1. **用户明确指示** - 最高优先级
2. **项目 steering 规范** - api-standards.md、structure.md 等
3. **技术栈最佳实践** - Vue 3、TypeScript、Ant Design Vue
4. **通用编程规范** - 代码可读性、可维护性

如果不同规范之间有冲突，询问用户如何处理。

## 特殊说明

### 关于中文

- 代码注释使用中文
- 类型定义的注释使用中文
- 与用户沟通使用中文
- 代码本身（变量名、函数名）使用英文

### 关于依赖

- 不要随意引入新的 npm 包
- 优先使用项目中已有的库
- 如果需要新依赖，先询问并说明理由

### 关于测试

- 如果用户没有要求，不主动编写测试代码
- 如果需要测试，询问测试范围和方式
